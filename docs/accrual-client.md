# HTTP-клиент системы расчёта (accrualclient)

Пакет `pkg/accrualclient` — это тонкая обёртка над **внешней системой расчёта начислений**.  
Она живёт отдельно от нашего сервиса (адрес задаётся в конфиге, например `http://localhost:8082`).

Клиент **не начисляет баллы сам** и **не ходит в нашу БД**. Он только делает HTTP-запрос «какой статус и сколько баллов у заказа?» и возвращает ответ в виде Go-структур и ошибок.

---

## Зачем он нужен

Когда пользователь загружает заказ, мы сохраняем его у себя со статусом `NEW`.  
Дальше **воркер** периодически спрашивает внешнюю систему: «этот заказ уже обработан? сколько начислить?»

За эти запросы отвечает `AccrualClient` и метод **`GetOrderInfo`**.

```
AccrualWorkerPool  →  accrualclient.GetOrderInfo  →  GET /api/orders/{number}
```

HTTP-хендлер пользователя клиент **не вызывает** — только воркер (через `PollOrder` в `internal/order/accrual.go`).

---

## Что делает GetOrderInfo

По номеру заказа отправляет:

```
GET {Address}/api/orders/{number}
```

и разбирает ответ:

| HTTP-код   | Что значит                                            | Что возвращает клиент              |
|------------|-------------------------------------------------------|------------------------------------|
| **200**    | Заказ найден, есть статус (и иногда сумма начисления) | `OrderInfo`                        |
| **204**    | Заказ ещё не зарегистрирован во внешней системе       | `ErrNotFound`                      |
| **429**    | Слишком много запросов                                | `RateLimitedError` + `Retry-After` |
| **500**    | Ошибка на стороне внешней системы                     | `ErrServer`                        |
| **другой** | Неожиданный ответ                                     | `UnexpectedStatusError`            |

Пример успешного ответа (200):

```json
{
  "order": "79927398713",
  "status": "PROCESSED",
  "accrual": 500
}
```

Поле `accrual` может отсутствовать — это нормально.

### Статусы во внешней системе


| status       | Смысл                                              |
|--------------|----------------------------------------------------|
| `REGISTERED` | Заказ зарегистрирован, начисление ещё не посчитано |
| `PROCESSING` | Расчёт в процессе                                  |
| `INVALID`    | Начисления не будет (финальный статус)             |
| `PROCESSED`  | Начисление посчитано (финальный статус)            |

Перевод этих статусов во внутренние (`orders.status` в нашей БД) делает `order.MapAccrualStatus`, не клиент.

## Ретраи

Клиент **сам повторяет запрос**, если произошёл **кратковременный сбой**.  
Для вызывающего кода ничего не меняется: по-прежнему один вызов `GetOrderInfo`.

Под капотом используется [hashicorp/go-retryablehttp](https://github.com/hashicorp/go-retryablehttp).

### Когда повторяем

- **Сеть** — таймаут, обрыв соединения, сервис временно недоступен.
- **HTTP 500** — внутренняя ошибка внешней системы.

Между попытками — пауза с backoff (сначала короче, потом дольше, в пределах настроек).

### Когда не повторяем

- **200** и **204** — это уже осмысленный ответ, повторять бессмысленно.
- **429** — лимит запросов; паузу выставляет **воркер** (`setRateLimit` по `Retry-After`), а не клиент.
- **Остальные коды** — повтор не поможет.

### Настройки по умолчанию

| Параметр              | Config         | Значение |
|-----------------------|----------------|----------|
| Число повторов        | `RetryMax`     | 3        |
| Мин. пауза            | `RetryWaitMin` | 500ms    |
| Макс. пауза           | `RetryWaitMax` | 5s       |
| Таймаут одной попытки | `Timeout`      | 5s       |

Если все попытки на 500 исчерпаны, клиент вернёт `ErrServer`.

## Конфигурация и создание

Клиент создаётся в `cmd/gophermart/di/app.go`:

```go
accrualClient := accrualclient.New(accrualclient.Config{
    Address: cfg.Accrual.Address,
})
```

Адрес внешней системы — `accrual.address` / `ACCRUAL_SYSTEM_ADDRESS`.

Интерфейс **`Provider`** с методом `GetOrderInfo` нужен, чтобы в тестах подменять клиент моком.

---

## Связь с воркером

Клиент не решает, **что делать** с ответом — только доставляет его.  
Логика «обновить заказ / подождать / повторить позже» — в `PollOrder` и `AccrualWorkerPool`:

| Ошибка / ответ клиента | Действие воркера |
|------------------------|------------------|
| 200 + `OrderInfo` | Обновить заказ в БД, при `PROCESSED` — начислить баллы |
| `ErrNotFound` (204) | Ничего не менять, scanner повторит позже |
| `RateLimitedError` (429) | Пауза для всех воркеров, задача снова в очередь |
| `ErrServer` / сеть | Лог, scanner повторит позже |

**Два уровня повторов:**

1. **Клиент** — сразу, в течение секунд (сеть, 500).
2. **Воркер / scanner** — позже, по расписанию (остальные случаи, следующий цикл опроса).
